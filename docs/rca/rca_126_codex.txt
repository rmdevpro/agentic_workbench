• Root Cause

  Both bugs come from the same design assumption: Blueprint treats Claude’s JSONL file and Claude session ID as if they were the universal session source of truth.

  For Gemini and Codex, that assumption is false.

  - Sidebar metadata is mostly built from Claude-only parsing, so Gemini/Codex sessions fall back to DB defaults and 0 messages.
  - Reopen/resume logic has no stored Gemini/Codex CLI session ID, so it falls back to “latest” / “last”, which is inherently wrong when multiple sessions exist.

  1. Metadata Read Paths

  - session-utils.parseSessionFile() reads only Claude-style .jsonl transcripts and extracts name, timestamp, messageCount, and model. It is the only real metadata
    parser in the session stack. session-utils.js:17
    Wrong for Gemini/Codex: it only understands Claude’s file location and schema (user / assistant JSONL). It does not read Gemini chat JSON or Codex rollout JSONL at
    all.
  - routes.buildSessionList() builds sidebar rows by calling parseSessionFile(join(sessDir, \${s.id}.jsonl`))` for every session, then falls back to DB fields.
    routes.js:280
    Blueprint’s DB ID, not Gemini/Codex transcript identity. For non-Claude sessionsfileMetais usuallynull`, so:
      - messageCount becomes 0
      - timestamp falls back to sessions.updated_at
      - name falls back to DB name
      - only model gets a separate non-Claude hack
  - routes._getNonClaudeModel() tries to read Gemini/Codex model names from disk. routes.js:219
    Wrong for Gemini: it scans every file under ~/.gemini/tmp/*/chats/*.json and returns the first message with a model, not the model for the current Blueprint
    session.
    Wrong for Codex: it recursively scans every file under ~/.codex/sessions/**/*.jsonl and returns the first turn_context.payload.model, again not tied to the current
    session.
    Extra bug: the result is cached by Blueprint session ID even though lookup is global, so a wrong model can stick. routes.js:216
  - GET /api/state backfills DB sessions only from Claude JSONLs by scanning safe.findSessionsDir(projectPath). routes.js:557
    Wrong for Gemini/Codex: their transcript files are never scanned here, so DB rows are never refreshed from actual Gemini/Codex conversation files.
  - mcp-tools.listSessions() lists sessions by scanning only Claude’s sessions dir and calling parseSessionFile(). mcp-tools.js:480
    Wrong for Gemini/Codex: same Claude-only assumption, so MCP listing is also blind to non-Claude metadata.
  - session-utils.searchSessions() searches only Claude JSONLs and uses cached Claude metadata for names. session-utils.js:113
    Wrong for Gemini/Codex: those sessions are unsearchable through this path because their files are never read.
  - Resume-prompt helpers also assume Claude-only transcript storage:
      - POST /api/sessions/:sessionId/session mode=resume reads only Claude JSONL. routes.js:1317
      - mcp-tools action=resume does the same. mcp-tools.js:233
        Wrong for Gemini/Codex: these helpers cannot read non-Claude conversation tails.
  - Existing non-Claude parsers already exist in qdrant-sync.js:
      - parseGeminiSession() qdrant-sync.js:359
      - parseCodexSession() qdrant-sync.js:391
        This is important because the repo already knows how to parse those file formats, but the sidebar/session stack does not reuse that logic.

  2. Resume Paths

  - Main reopen path from UI:
      - UI click calls POST /api/sessions/:id/resume. public/index.html:1146
      - Server route recreates tmux if dead. routes.js:767
        Wrong for Gemini: uses ['--resume'] with no ID. routes.js:796
        Wrong for Codex: uses ['resume', '--last']. routes.js:799
        Result: opens most recent / last conversation, not the specific one the user clicked.
  - Restart path from UI:
      - UI restart button calls POST /api/sessions/:id/restart. public/index.html:1116
      - Server restart route kills tmux and recreates it. routes.js:1351
        Wrong for Gemini/Codex: same no-ID resume args. routes.js:1364
        Additional bug: it uses db.getSession(sessionId) then reads session.project_name, but getSession() does not join project data, so project_name is undefined and
        it falls back to WORKSPACE. routes.js:1358
        For non-Claude CLIs, where “latest/last” can be cwd-sensitive, that makes the wrong-session problem even worse.
  - MCP reopen path:
      - mcp-tools.ensureSessionTmux() recreates tmux when needed. mcp-tools.js:132
      - Used by MCP connect and restart. mcp-tools.js:167
        Wrong for Gemini/Codex: same ['--resume'] / ['resume', '--last'] bug. mcp-tools.js:138
  - Session creation is the upstream cause:
      - Non-Claude sessions are created with a random Blueprint UUID up front. routes.js:675
      - session-resolver.resolveSessionId() explicitly does nothing for non-Claude CLIs. session-resolver.js:19
        Wrong for Gemini/Codex: Blueprint never records the CLI’s real session ID at all, so later exact resume is impossible.
        The same pattern exists in MCP session creation. mcp-tools.js:149
  - safe.tmuxCreateCLI() is not the root bug; it just launches whatever args callers pass. safe-exec.js:93
    For Gemini/Codex it faithfully launches the wrong resume command because callers never provide the exact CLI session ID.

  3. What Is Specifically Wrong For Gemini And Codex

  - Gemini metadata:
      - No code path reads the clicked session’s specific ~/.gemini/tmp/<project>/chats/session-*.json file for count/name/timestamp.
      - Only model is read, and even that is read by global first-match scan, not by session mapping.
  - Codex metadata:
      - No code path reads the clicked session’s specific ~/.codex/sessions/<rollout>.jsonl for count/name/timestamp.
      - Model is also read by global first-match scan, not by session mapping.
      - state_5.sqlite is not used to map Blueprint session -> Codex internal session/thread ID.
  - Gemini resume:
      - Reopen uses gemini --resume without an ID, so it resumes “latest”, not the clicked session.
  - Codex resume:
      - Reopen uses codex resume --last, so it resumes the last session, not the clicked one.
  - Shared architectural bug:
      - Blueprint DB session ID is not the Gemini/Codex CLI session ID.
      - There is no persisted mapping from Blueprint session row to Gemini sessionId or Codex resume ID.

  4. Recommended Fix Approach

  1. Add a per-CLI session adapter layer instead of reusing Claude parsing everywhere.
      - readSessionMeta(session) -> { name, timestamp, messageCount, model }
      - resolveCliSessionId(session) -> exact CLI resume ID
      - resumeArgs(session) -> exact args for that CLI
  2. Persist non-Claude identity in DB.
      - Add fields like cli_session_id, transcript_path, maybe meta_file_mtime / meta_file_size.
      - Claude: existing real session ID already serves this role.
      - Gemini: store the JSON file path and sessionId from that file.
      - Codex: store the exact ID required by codex resume <id> plus the rollout/transcript path.
  3. Replace sidebar metadata assembly with CLI-specific parsing.
      - Claude: keep parseSessionFile().
      - Gemini: parse the matching chat JSON for:
          - name from first user message or saved title
          - timestamp from last message
          - messageCount from actual message array
          - model from latest assistant/gemini message with model
      - Codex: parse the matching rollout JSONL for:
          - name from first user/input turn
          - timestamp from latest relevant entry
          - messageCount from actual user/assistant turns
          - model from turn_context.payload.model or equivalent latest turn context
  4. Stop using _getNonClaudeModel() as a global scan.
      - It should be deleted or rewritten to operate only on the mapped file for the current session.
  5. Replace every non-Claude resume call with exact-ID resume only.
      - routes.js /resume
      - routes.js /restart
      - mcp-tools.js ensureSessionTmux
        Use:
      - Gemini: ['--resume', cli_session_id]
      - Codex: ['resume', cli_session_id]
        Never ['--resume'] and never ['resume', '--last'].
  6. Fix routes.js restart cwd bug.
      - Use db.getSessionFull(sessionId) instead of db.getSession(sessionId) in /restart, or pass project path explicitly.
  7. Reuse existing parser knowledge from qdrant-sync.js.
      - The repo already has working Gemini/Codex transcript parsers. Those should be moved into shared session parsing utilities instead of duplicating logic.
  8. Add tests that currently do not exist.
      - No current resume tests cover Gemini/Codex exact-ID behavior. tests/mock/routes.test.js:1248
      - Add tests for:
          - sidebar shows non-zero message counts for seeded Gemini/Codex files
          - exact CLI session ID is stored
          - /resume and /restart pass exact ID args
          - /restart uses correct project cwd

  The short version is: Claude has a real transcript parser and a real session-ID mapping; Gemini and Codex have neither. That is why metadata falls back to 0, and why
  reopen falls back to “latest/last”.
