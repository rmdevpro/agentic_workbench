  Root Cause Analysis

  Bug 1: Gemini and Codex sessions show 0 messages

  Code path: buildSessionList() at routes.js:280-300

  const jsonlPath = join(sessDir, `${s.id}.jsonl`);
  const fileMeta = await sessionUtils.parseSessionFile(jsonlPath);

  Root cause: buildSessionList() constructs a path <sessDir>/<sessionId>.jsonl for every session regardless of CLI type. This is the Claude JSONL path
  (~/.claude/projects/<hash>/<id>.jsonl). Gemini and Codex don't write files there. Their data lives at:

  - Gemini: ~/.gemini/tmp/<project>/chats/session-*.json
  - Codex: ~/.codex/sessions/<rollout>.jsonl

  Since no file exists at the constructed path, parseSessionFile() returns null (ENOENT at line 94), so fileMeta?.messageCount is undefined, which falls back to || 0.

  The model extraction (_getNonClaudeModel()) does scan the correct directories, but there is no equivalent function that scans Gemini/Codex directories for message
  counts or names. The qdrant-sync.js file has parsers (parseGeminiSession, parseCodexSession) that correctly read messages from those paths, but they're only used for
  vector indexing — never called by buildSessionList().

  Bug 2: Resume opens wrong conversation for Gemini/Codex

  Code path: POST /api/sessions/:sessionId/resume at routes.js:767-815

  if (cliType === 'gemini') {
    resumeArgs = ['--resume'];       // resumes most recent, NOT the specific session
  }
  if (cliType === 'codex') {
    resumeArgs = ['resume', '--last']; // resumes last session, NOT the specific one
  }

  Root cause: The code passes --resume (Gemini) and resume --last (Codex) without any session identifier. Both CLIs support resuming a specific conversation:

  - Gemini: --resume <session-file-path> or similar session specifier
  - Codex: resume <rollout-id> (the rollout directory name)

  But Blueprint has no mapping from its internal sessionId (a UUID assigned at creation time) to the CLI's native session identifier (Gemini's chat file path, Codex's
  rollout directory name). The TODO at line 794 acknowledges this.

  Recommended Fix

  Fix 1: Message counts

  Add a parseNonClaudeSession(session) function that, given a session's cli_type and id:

  1. Gemini: Scan ~/.gemini/tmp/*/chats/*.json, find the file corresponding to this session (you'll need a mapping — see Fix 2), parse its .messages array, count
  user/model turns, extract first user message as name.
  2. Codex: Scan ~/.codex/sessions/, find the rollout directory for this session, parse its .jsonl, count response_item entries by role.
  3. In buildSessionList(), branch on cli_type:
  if (s.cli_type === 'claude') {
    fileMeta = await sessionUtils.parseSessionFile(join(sessDir, `${s.id}.jsonl`));
  } else {
    fileMeta = await sessionUtils.parseNonClaudeSession(s);
  }

  The parsers already exist in qdrant-sync.js — extract them into session-utils.js and reuse.

  Fix 2: Resume with correct session

  Store the native session identifier at creation time. When a new Gemini/Codex session is created:

  1. After tmuxCreateCLI(), watch for the native session file to appear (similar to how session-resolver.js watches for Claude's JSONL).
  2. Store the native path/ID in a new DB column (native_session_id on the sessions table).
  3. On resume, use:
    - Gemini: the specific chat file path or internal Gemini session arg
    - Codex: resume <rollout-id> instead of resume --last

  This mirrors what session-resolver.js already does for Claude (resolving new_* temp IDs to real UUIDs), but adapted for Gemini/Codex native identifiers.
