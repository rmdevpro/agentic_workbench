✦ Based on a comprehensive review of the Blueprint Test Runbook (tests/blueprint-test-runbook.md) and the application source code (public/index.html), here is the
  detailed assessment.

  1. COMPLETENESS: UI Function Coverage
  The runbook covers core "happy path" workflows but omits several secondary UI features that exist in the codebase:
   * Task Detail Modal: While tests/browser/task-tree.spec.js covers this, the runbook (intended as the master manual/automated guide) lacks steps for opening the task
     detail modal, editing notes/history, and verifying the visible class toggle (Line 261 in index.html).
   * Home Buttons: The "Home" navigation buttons for both the Task Tree (autoNavigateTaskTree) and File Browser (autoNavigateFileTree) on lines 530 and 537 of
     index.html have no corresponding test cases in the runbook.
   * Session Resolution (Temp IDs): The logic for migrating temporary IDs (new_*) to real session UUIDs (Lines 903–928 in index.html) is critical for UI stability but
     is not explicitly verified in the runbook.
   * Theme/Logo Toggling: Visual verification of the theme-swapping logo (.logo-light vs .logo-dark on line 51) is missing from the Appearance tests (USR-04).

  2. CLI PARITY: Claude vs. Gemini/Codex
  There is a significant gap in interactive testing between Claude and the other two CLI types:
   * Claude CLI (Phase 5): Features 21 detailed interaction tests (CLI-01 to CLI-21), covering slash commands (/help, /status), tool usage, and interruption.
   * Gemini/Codex Gaps:
       * Interaction: No matching tests for interactive prompts within the Gemini or Codex terminals.
       * Tools: Tool usage (CLI-18) is only verified for Claude.
       * Commands: Parity for slash commands or their equivalents in Gemini/Codex is not addressed.
       * Multi-line: The multiline editor behavior specific to Gemini/Codex (noted in using-other-clis-via-tmux.md as requiring extra Enters) is not tested in the
         runbook.
   * Existing Coverage: Only session creation (SESS-04, SESS-07), basic persistence (SESS-05), and type indicators (SESS-06) are covered for Gemini/Codex.

  3. REAL BROWSER TESTING: API vs. UI Interaction
  Several "UI tests" rely on injecting data via browser_evaluate rather than simulating user input, which risks missing bugs in the UI's event handling layers:
   * Terminal Input: Many CLI- tests (e.g., CLI-11, CLI-12, CLI-13) use tabs.get(id).ws.send(...) to send text. This bypasses XTerm.js key event handling and the UI's
     input buffer logic.
   * Settings/Search: CORE-06 (filter select) and FEAT-08 (search input) use browser_evaluate to set values and dispatch events manually rather than using
     page.selectOption or page.fill.
   * Pure API Tests: FEAT-20 (Global Search) is performed entirely via a fetch call inside browser_evaluate, testing the backend but not the search UI's rendering of
     results.

  4. CONVERSATION WORKFLOW: Create → Talk → Close → Reopen
   * Claude: Verified via CLI-19. It tests buffer persistence after a browser_refresh and tab reopening.
   * Gemini/Codex: INCOMPLETE. SESS-05 (Gemini Persistence) only verifies that the session remains in the /api/state list. It does not verify that a user can reopen the
     tab and see the previous conversation history/terminal buffer. There is no matching persistence test for Codex.

  5. ADEQUACY: Overall Assessment
  The runbook is a strong baseline for functional verification but has "blind spots" in state management and cross-CLI consistency:
   * Categories of Bugs that would slip through:
       * Race Conditions: The complex new_* to UUID migration logic could fail under high latency, leading to "Identity Swaps" (as noted in rca-91-findings.txt) which
         are not explicitly stress-tested here.
       * UI Input Bugs: Since ws.send is used for many terminal tests, bugs in how the UI captures keyboard shortcuts or multi-byte characters might be missed.
       * Terminal Capture Failures: If the tmux capture logic fails specifically for Gemini's multiline output, SESS-05 would still pass because the session exists, but
         the user would see an empty terminal.
       * Platform Discrepancies: Path handling (Windows vs Linux) is noted as a risk in recent RCAs, but the runbook lacks explicit "Cross-platform path" verification
         tests (e.g., creating files with backslashes or deep nesting).
